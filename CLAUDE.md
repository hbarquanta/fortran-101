# Fortran 101 Workshop — Rules for Claude

This repo is a beginner's live Fortran workshop (user has never written
Fortran before, taught primarily by a human teacher; I'm secondary
support). These rules govern anything I write here.

## Golden rule: as simple as possible

Every example must teach exactly **one concept**. When in doubt, cut,
don't add. Concretely:

- Brief, good comments are welcome — one short line explaining a
  non-obvious bit (e.g. why a type was guessed, what an attribute does).
  Never a multi-line block or a comment restating what the code already
  says.
- No labeled/decorated print statements — a single `print *, a, b, c` is
  fine; don't wrap each value in its own `print *, 'label:', value`.
- No extra variables, branches, or types beyond what the one concept
  needs. If two things are being contrasted (e.g. function vs
  subroutine), that means **two separate files**, not one file showing
  both.
- Don't mix an "optional" variant with the base case unless the variant
  *is* the concept (e.g. don't declare both `integer` and
  `integer(kind=8)` in the same file just to show kind exists — pick one
  spot to show it).
- If it still feels like more than one idea, split it.
- Prefer explicit assignment (`a = .true.` after `logical :: a`) over
  inline declaration initializers (`logical :: a = .true.`) — the
  latter implicitly gives the variable the `save` attribute, a subtlety
  beginners shouldn't hit by accident. Only use inline initializers when
  that syntax itself is the concept being taught (e.g. character literal
  declarations).

If unsure whether something counts as "too much", err toward less and
let the user ask for more.

## File naming & numbering

- Every file in `src/` gets a two-digit numeric prefix matching its
  section number in `docs/README.md`: `NN_name.f90` (or `.f` for
  fixed-form).
- Multiple files can share the same number when they belong to the same
  doc section (e.g. `05_function_example.f90` and
  `05_subroutine_example.f90`).
- `bin/` binaries are generated by the Makefile and mirror the `src/`
  filename exactly — never hardcode binary names in the Makefile.

## Build system

- Every program has `implicit none` as its first line — **except**
  `src/17_without_implicit_none.f90`, which deliberately omits it to
  show what type-guessing looks like. Don't "fix" that file.
- The Makefile is generic: it globs `src/*.f90` and `src/*.f` and builds
  a matching `bin/<name>` for each. Don't add per-file rules — if a new
  extension needs support, extend the wildcard/pattern rules, not a
  one-off target.
- After adding or changing a `src/` file: rebuild with `make` and run the
  binary to confirm the output, in the same turn.

## Documentation

- All concept documentation lives in a **single file**: `docs/README.md`.
  Do not create additional files under `docs/`.
- Sections are numbered sequentially in the order concepts were learned,
  matching the `src/` file numbers. Update the "Contents" list at the
  top whenever a section is added.
- Each section: a short explanation, one code block, a link to the
  source file, and the build/run command (`make && ./bin/NN_name`).
- Add the doc section in the same turn the source file is added — don't
  let it drift.

## When the user says "too complicated" / "overengineering"

Treat it as: cut harder, don't rephrase. Remove comments, remove
per-line labeled prints, remove any variable or type not strictly
needed, and split multi-concept files into separate single-concept
files.
